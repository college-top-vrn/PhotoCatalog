# Полное Техническое Задание  на разработку библиотеки каталогизации фотографий


## 1. Назначение документа и описание проекта

**Назначение:** Документ описывает архитектуру, функциональные требования и структуру компонентов для системы каталогизации фотографий.

**Суть проекта:** Разработка ядра (библиотеки) для управления коллекцией фотографий с использованием концепции "виртуальных альбомов". Физические файлы хранятся в единой директории, а иерархия папок, альбомов и тегов формируется исключительно на логическом уровне в базе данных.

## 2. Архитектурные требования и стек технологий

Система проектируется с соблюдением принципов **Чистой Архитектуры, **SOLID** и **Богатой Доменной Модели **.

- **Основная платформа:** .NET 10
    
- **Язык программирования:** C# 13
    
- **База данных:** SQLite
    
- **Доступ к данным:** **Dapper**. Используется как легковесный ORM для выполнения SQL-запросов и маппинга результатов в доменные объекты, обеспечивая баланс между производительностью, контролем над SQL и удобством разработки.
    
- **Тестирование:** xUnit (Unit и интеграционные тесты), ArchUnitNET (архитектурные тесты).
    
- **Инверсия зависимостей (DIP):** Внутренние слои (`Domain`) абсолютно независимы от внешних (`Infrastructure`, `Application`, сторонние библиотеки). Взаимодействие происходит строго через контракты (интерфейсы).
    

## 3. Функциональные требования

### 3.1. Управление физическими файлами

- **Импорт:** Система принимает путь к файлу, вычисляет его хэш, извлекает метаданные (размер, ширина, высота) и копирует/перемещает его в единую целевую директорию хранилища.
    
- **Синхронизация:** Система проверяет наличие физического файла по сохраненному пути и помечает записи в БД как "осиротевшие", если файл был удален средствами ОС.
    
- **Физическое удаление:** При явном удалении фотографии из системы файл удаляется с диска, а все виртуальные связи в БД уничтожаются каскадно.
    

### 3.2. Управление виртуальной иерархией

- **Виртуальные папки:** Создание, переименование, удаление и поддержка вложенности (папка внутри папки). На диске эти папки не создаются.
    
- **Виртуальные альбомы:** Создание альбомов внутри папок или в корне виртуального дерева.
    
- **Управление контентом:** Добавление фотографии в альбом (создание логической связи). Одна фотография может одновременно находиться в неограниченном количестве альбомов. Удаление фото из альбома удаляет только связь, физический файл остается нетронутым.
    

### 3.3. Тегирование и поиск

- **Тегирование:** Создание строковых тегов и привязка их к фотографиям.
    
- **Поиск и фильтрация:** Получение списка фотографий по пересечению условий:
    
    - Нахождение в определенном виртуальном альбоме или папке.
        
    - Наличие всех или хотя бы одного из указанных тегов.
        

## 4. Структура проекта (Слои)

Директории (папки) в решении должны именоваться строго в **PascalCase**. Проект разделен на три основных слоя:

### 4.1. PhotoCatalog.Domain (Ядро)

Не имеет никаких внешних зависимостей. Содержит чистую бизнес-логику.

- **Entities:** `Photo`, `Album`, `Folder`, `Tag`. Инкапсулируют состояние (используются `private` или `init`-only сеттеры). Логика изменения состояния находится внутри самих классов (например, `album.AddPhoto(photo)`, `folder.MoveTo(newParentFolder)`).
    
- **Value Objects:** Неизменяемые объекты без идентичности (например, `Dimensions` для габаритов фото).
    
- **Exceptions:** Доменные исключения для контроля бизнес-правил и инвариантов (например, `PhotoAlreadyInAlbumException`, `InvalidTagFormatException`).
    
- **Interfaces:** `IPhotoRepository`, `IFolderRepository`, `IFileStorage`. Все контракты живут здесь.
    

### 4.2. PhotoCatalog.Application (Сценарии использования)

Зависит только от `Domain`.

- **Use Cases (Services):** Обработчики команд, реализующие конкретные действия пользователя (например, `ImportPhotoUseCase`, `AddTagUseCase`).
    
- **DTO:** Простые структуры данных для возврата результатов на уровень представления, чтобы не отдавать наружу доменные сущности напрямую.
    

### 4.3. PhotoCatalog.Infrastructure (Реализация)

Зависит от `Domain` и `Application`.

- **Data Access:** Реализация репозиториев для SQLite с использованием **Dapper**. Включает написание оптимизированных SQL-запросов и логику маппинга плоских табличных данных в богатые доменные объекты (например, маппинг связей Many-to-Many).
    
- **File System:** Реализация `IFileStorage` через классы `System.IO`.
    

## 5. Проектирование базы данных SQLite

Схема спроектирована для эффективной работы с реляционными связями:

- `Photos`: Id, RealPath (UNIQUE), FileHash, Width, Height, AddedAt.
    
- `Folders`: Id, Name, ParentFolderId (FK -> Folders.Id).
    
- `Albums`: Id, Name, FolderId (FK -> Folders.Id).
    
- `Tags`: Id, Name (UNIQUE).
    
- `AlbumPhotos`: AlbumId, PhotoId (Composite PK, ON DELETE CASCADE).
    
- `PhotoTags`: PhotoId, TagId (Composite PK, ON DELETE CASCADE).
    

**Обязательное требование:** При каждом открытии соединения в `Infrastructure` (до выполнения команд) необходимо выполнять `PRAGMA foreign_keys = ON;` для активации ссылочной целостности SQLite.

## 6. Нефункциональные требования к коду и тестированию

- **Стиль кода:** Строгое соблюдение официального Style Guide от Microsoft (C#).
    
- **Архитектурные тесты (ArchUnitNET):** Наличие автоматических тестов, проверяющих направления зависимостей (`Domain` не знает про `Infrastructure`), правильность неймспейсов (соответствуют папкам в PascalCase) и расположение контрактов.
    
- **Unit-тесты (xUnit):** Покрытие доменных сущностей и `UseCase` классов (с использованием стабов/моков для репозиториев).
    
- **Обработка бизнес-ошибок:** Использование кастомных доменных исключений (Domain Exceptions) для ситуаций, нарушающих бизнес-логику. Перехват этих исключений осуществляется на уровне `Application` или Presentation.
    
- **Защита инвариантов:** Доменная модель не должна позволять перевести систему в невалидное состояние (например, создание циклической зависимости в иерархии папок, добавление пустых тегов). Валидация происходит внутри `Domain`.
    

## 7. Этапы разработки

1. **Domain & Testing:** Создание сущностей (`Photo`, `Album`, `Folder`) с инкапсуляцией. Написание кастомных исключений и Unit-тестов на логику изменения состояния.
    
2. **Архитектурные тесты:** Настройка ArchUnitNET для фиксации слоев, контроля зависимостей и правил именования.
    
3. **Скелет БД:** Создание DDL-скриптов для генерации файла SQLite со всеми таблицами и настроенными каскадными удалениями.
    
4. **Data Access (Dapper):** Реализация CRUD-операций и сложных выборок в `Infrastructure`. Настройка Dapper для корректного маппинга связей Many-to-Many (например, сборка `Photo` с коллекцией `Tags` через `QueryMultiple` или маппинг словарей).
    
5. **Use Cases:** Реализация прикладного слоя (`Application`) и связывание логики файловой системы (`System.IO`) с репозиториями базы данных.
    
6. **Интеграционное тестирование:** Запуск In-Memory SQLite (или генерация тестового файла БД) и проверка полного цикла работы приложения от импорта фото до поиска по тегам.
