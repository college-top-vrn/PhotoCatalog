Руководство по Result Fluent Api
---

## 1. Концепция: Двухпутная железная дорога

В основе лежит метафора **Railway Oriented Programming**. Представьте, что выполнение функции — это движение поезда.

- **Путь Успеха (Success Track):** Поезд едет прямо, передавая данные от одной станции (метода) к другой.

- **Путь Ошибки (Failure Track):** Если на любой станции происходит сбой, стрелка переключается. Поезд переходит на параллельный путь. На этом пути все последующие «станции успеха» игнорируются — поезд просто везет объект ошибки до самого конечного терминала.

## 2. Создание результата: Явность против Не явности

Класс `Result<T>` позволяет создавать объект двумя способами.

### Неявное приведение (Implicit)

Код выглядит лаконично, так как компилятор сам догадывается, во что обернуть значение.


```cs
public Result<int> GetStock() => 10; // Авто-упаковка в Success
public Result<int> GetError() => Error.NotFound; // Авто-упаковка в Failure
```

### Фабричные методы (Explicit) — **Рекомендуемый подход**

Мы настоятельно рекомендуем использовать `Result.Success()` и `Result.Failure()`.

**Почему?** Это делает намерения программиста кристально ясными. Читая код, ваш коллега (или вы через месяц) сразу увидит логическое ветвление, не заглядывая в сигнатуры методов.


```cs
public Result<decimal> CalculateDiscount(Customer customer)
{
    if (customer.IsVip)
        return Result<decimal>.Success(0.15m); // Явно: это успех

    return Result<decimal>.Failure(DiscountErrors.NotEligible); // Явно: это ошибка
}
```

## 3. Как работают методы (Внутренняя механика)

Давайте заглянем «под капот» расширений, чтобы понять, как они управляют потоком.

### `Then` (Связывание)

**Логика:** «Если всё хорошо — делай следующее, если нет — забудь».

Внутри метода стоит простая проверка: если текущий результат `IsFailure`, метод `nextStep` даже не будет вызван. Ошибка просто пробрасывается дальше. Это и есть та самая «стрелка» на железной дороге.

### `Transform` (Преобразование)

**Логика:** «Если всё хорошо — измени данные».

В отличие от `Then`, этот метод ожидает функцию, которая **не может** вернуть ошибку. Он берет значение, меняет его (например, из `User` делает `string` с его именем) и сам упаковывает обратно в `Result.Success`.

### `OnSuccess` / `OnFailure` (Наблюдатели)

**Логика:** «Посмотри на результат и, если он подходит, сделай что-то полезное со стороны».

Эти методы всегда возвращают тот же самый объект `Result`, который получили на вход. Они не могут изменить данные внутри контейнера. Это идеально подходит для логирования или отправки уведомлений.

## 4. Подробная шпаргалка по методам

|**Метод**|**Что делает «под капотом»?**|**Когда применять?**|
|---|---|---|
|**`Success(T)`**|Создает объект с `IsSuccess = true`.|Когда нужно вернуть данные.|
|**`Failure(E)`**|Создает объект с `IsSuccess = false` и ошибкой.|Когда бизнес-логика не может продолжаться.|
|**`Then`**|Проверяет `IsSuccess`. Если `true` — запускает следующую функцию.|Когда следующий шаг тоже может вернуть ошибку (например, запрос к БД).|
|**`Transform`**|Проверяет `IsSuccess`. Если `true` — меняет тип данных.|Для маппинга или извлечения свойств (например, `User -> UserDto`).|
|**`OnSuccess`**|Выполняет `Action`, если результат успешен.|Для логирования успеха или запуска триггеров.|
|**`OnFailure`**|Выполняет `Action`, если результат — ошибка.|Для записи ошибок в лог или телеметрию.|
|**`Finally`**|Принимает две функции и выполняет одну из них.|В самом конце, чтобы выйти из «контейнера» в UI или API.|

## 5. Полный пример: Система заказа еды

Посмотрите, как методы работают вместе, создавая читаемую цепочку без вложенных `if`.
```cs
public string ProcessPizzaOrder(int orderId)
{
    return _repository.GetOrder(orderId)           // 1. Получаем заказ (Result<Order>)
        .Then(order => _kitchen.Prepare(order))    // 2. Готовим (может не быть ингредиентов)
        .Then(pizza => _delivery.Assign(pizza))    // 3. Ищем курьера (может не быть свободных)
        .Transform(task => task.TrackingNumber)    // 4. Нам нужен только номер отслеживания
        .OnSuccess(track => _logger.Info(track))   // 5. Логируем успех
        .OnFailure(err => _logger.Error(err.Code)) // 6. Логируем ошибку, если она случилась на любом этапе
        .Finally(                                  // 7. Возвращаем ответ клиенту
            success: track => $"Ваш заказ в пути! Номер: {track}",
            failure: error => $"Не удалось оформить заказ: {error.Description}"
        );
}
```

### Почему использоватьтакая технику вычисления  лучше?

Если `_repository.GetOrder` вернет ошибку (заказ не найден), методы `Prepare`, `Assign` и `Transform` будут пропущены автоматически. Поезд сразу окажется на этапе `OnFailure` и `Finally`.
