# Создание ветки
В проекте используется методология Git Flow для упрощение разработки. В Git Flow стандартные типы веток, которые создаются от develop, используют определённые префиксы для категоризации по назначению.​  

Основные типы веток:
- feature/ — для разработки новых функций (например, feature/user-authentication, feature/login-page)​;
- improvement/ — для улучшений и оптимизаций (например, improvement/refactor-auth)​;
- bugfix/ — для исправления багов​.

# Создание Pull requests
Маленькие и направленные PR:
- Один PR = одна задача / одна логическая фича или баг‑фикса.
- Не мешать в один PR: дизайн, бизнес‑логику, тесты и рефакторинг всего проекта.

Правильное название и описание PR
Заголовок:

Короткий, но информативный:
- Add JWT auth for API endpoints
- Fix race condition in email queue
  
Описание:
- Цель изменения: что делает PR и зачем.
- Влияние: как изменение затрагивает пользователей, API, БД, миграции.
- Ссылка на тикет/issue (например, Closes #123).
- Инструкции для тестирования, если нужно (например, curl‑примеры, сценарии).

Ревью и командный процесс
- Всегда назначать 1–2 ревьюера (через Assignees или Reviewers).
- Не мержить PR самому, если этого запрещает политика (обычно нужно approval от ревьюера).
- Реагировать на комментарии:
    либо править
    либо мотивированно отвечать, почему не правишь.

Пост‑мердж практики
- После мержа:
    удалить временную ветку;
    обновить main/develop локально и убедиться, что всё работает.
  
# Проектирование классов
Исполнитель должен перед реализацией спроектировать основные сущности, используя инструмент Mermaid.

Требования проектирования:
- Использование пространства имён;
- Использования типов связей;
- Использования обозначения модификаторов поведения и доступов для классов и членов класса;
- Сохранить код работы в markdown-файле с названием, отражающее то, что спроектированно в диаграмме;
- Загружать файл в путь `docs/directories/`.
  
# Создание файлов и директорий
Именование директории (папки):
- Каждое слово начинается с заглавной буквы, никаких пробелов или подчеркиваний;
- Название пишется во множественном числе и должно отражать содержимое (то, что оно в себе содержит).

Именование файла:
- Каждое слово начинается с заглавной буквы, никаких пробелов или подчеркиваний;
- Название отражает название класса.

# Правила оформления тест-кейсов

Каждый тест-кейс должен содержать::
- Заголовок: Укажите участников (тестировщик, разработчик) и тестируемый объект (метод, класс и т. д.).
- Описание: Что именно тестируется и ожидаемый результат.
- Результат: Явно укажите статус — "Прошёл" или "Не Прошёл" и написать сам результат.
- Доказательства: Прикрепите скриншот результата теста (консоль, IDE, Postman и т.д.).

Пример тест-кейса:

Тест-кейс: Иванов И.И. (тестер) + Петров П.П. (разработчик)
UserService.GetUserById(id: 123)

Ожидаемый результат: Возвращает User с ID=123 или null

СТАТУС: Прошёл

Результат: Вернул  User с ID=123.

Дата: 17.04.2026

[Скриншот результата]
